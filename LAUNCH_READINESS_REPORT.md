# Аудит готовности платформы к запуску

Проект: платформа учета бань  
Дата аудита: 2026-05-08

## Итоговая оценка готовности

**Текущая готовность: 55%**

### Почему не выше
- Бизнес-функции уже собраны (бронирование, документы, финансы, сотрудники, роли, поддержка), но есть критичные блокеры по **безопасности**, **целостности данных** и **эксплуатационной готовности**.
- Часть важной защиты реализована на фронте, но не дублируется строго на backend.
- Недостаточно автоматизированных тестов и релизной инфраструктуры для уверенного production-запуска.

---

## Что уже хорошо (сильные стороны)

- Разделение по доменным модулям на backend и frontend уже зрелое.
- Реализованы базовые роли/права и в целом рабочая модель доступа.
- Финансовый модуль с операциями приход/расход и счетами организации уже встроен в контур.
- Есть аудит-логи действий.
- Есть валидации в ключевых формах (бронирование, документы, реквизиты).

---

## Критические блокеры (P0 — обязательно до запуска)

## 1) Доступ и безопасность API
- Нужно гарантировать строгую server-side проверку прав на всех критичных endpoint'ах (не только на уровне UI).
- Нужно закрыть публичный доступ к чувствительным действиям и данным (документы, внутренние списки и т.д.).
- Нельзя допускать запуск с дефолтными fallback-секретами для JWT/сессий.

**Что сделать:**
- Пройти все admin/finance/documents/staff routes и везде поставить обязательную проверку пользователя и прав.
- Вынести проверки в единый декоратор/зависимость и использовать единообразно.
- Сделать fail-fast, если секреты не заданы окружением.

## 2) Целостность данных (финансы/склад/документы)
- Операции документов и связанных движений должны быть атомарными (одна транзакция на use-case).
- Нельзя допускать частичных записей при ошибке в середине операции.
- Обновление/удаление документов должно корректно отражать влияние на остатки и финансы по согласованным бизнес-правилам.

**Что сделать:**
- Перевести create/update/delete критичных документов на строгую транзакционную схему.
- Зафиксировать правила перерасчета склада и финансов при edit/delete.
- Добавить защиту от “исторических дыр” при изменении ключевых связей.

## 3) Минимальный quality gate перед релизом
- Сейчас недостаточно автотестов для production go-live.

**Что сделать:**
- Добавить минимум smoke/integration тесты:
  - auth/login/refresh/logout,
  - проверки прав на API,
  - бронирование + закрытие + влияние на документы/финансы,
  - документы прихода + влияние на склад/финансы.

## Важные улучшения (P1 — сразу после P0)

## 1) Миграции и схема БД
- Нужна единая стратегия миграций (без смешения “DDL в runtime” и отдельных ручных изменений).

**Что сделать:**
- Централизовать миграции.
- Проверку миграций включить в CI.

## 2) Операционная готовность
- Нужны health/readiness checks, базовая observability и стабильный релизный конвейер.

**Что сделать:**
- Добавить `/health` и `/readiness`.
- Включить структурные логи без чувствительных данных.
- Внедрить CI: lint + tests + migration check + build.

## 3) Логирование и DX
- Убрать лишние debug-print/console в production-path.

**Что сделать:**
- Централизовать логгер, уровни `info/warn/error`.
- Отключить шумящие логи в боевом режиме.

---

## Улучшения продукта (P2 — после стабилизации)

- Добавить e2e smoke-path для ключевых ролей.
- Расширить audit trail по финансовым операциям (документ-источник, счет, кто изменил, когда).
- Улучшить страницу фин.отчетов (срезы, экспорт, сверка периодов).
- Добавить сценарий “перевод между своими счетами” отдельным типом операции.
- Подготовить административный чек-лист закрытия дня/месяца.

## Что бы я еще добавил до запуска платформы

## 1) “Launch Gate” чек-лист (обязателен)
- Без прохождения чек-листа релиз запрещен.
- Пункты: миграции, smoke API, smoke UI, права, резервное копирование, rollback plan.

## 2) Резервирование и восстановление
- Регламент бэкапов БД + тест восстановления.
- Отдельно проверить восстановление финансовых и бронировочных данных.

## 3) Анти-регресс на критических потоках
- Набор тест-кейсов:
  - создание/редактирование/закрытие брони,
  - документ поступления,
  - влияние на финансы и склад,
  - фильтрация финансов по счету/периоду/типу.

## 4) Role hardening
- Таблица соответствия “роль -> разрешенные действия API”.
- Автотесты, которые гарантируют запрет лишних действий.

## 5) Предрелизная нагрузочная проверка (минимум)
- Проверить пики по спискам бронирований, финансов, документов.
- Проверить время ответа на типовых фильтрах и массовых выборках.

## Рекомендуемый план выхода в прод

## Этап 1 (1-3 дня): закрытие P0
- Доступы + безопасность + транзакционность + smoke тесты.

## Этап 2 (2-4 дня): P1
- Миграции, CI, health/readiness, прод-логирование.

## Этап 3 (2-5 дней): P2
- E2E, улучшения аудита/аналитики, UX улучшения.

После завершения Этапа 1 и успешного прогонa Launch Gate можно делать **ограниченный soft launch**.

## Финальный вывод

Продукт **близок к рабочему состоянию**, но перед запуском платформы нужно закрыть критические вещи по безопасности и целостности данных.  
С учетом текущего состояния: **55% готовности**.  
При фокусной работе по P0/P1 проект можно вывести в устойчивый прод-контур в короткий срок.

---

## Детальный план работ по этапам

Ниже — практический execution-план, который можно брать в спринт как есть.

## Этап 0: Подготовка (0.5 дня)

**Цель:** зафиксировать рамки релиза и не потерять фокус.

- Назначить ответственных:
  - backend owner,
  - frontend owner,
  - QA owner,
  - release owner.
- Создать доску задач: `P0 / P1 / P2`.
- Зафиксировать freeze на новые фичи до закрытия P0.

**Definition of Done:**
- есть единый список задач с приоритетами;
- у каждой задачи есть владелец и срок.

---

## Этап 1: Безопасность и права (P0, 1-2 дня)

**Цель:** убрать риски несанкционированного доступа.

### Backend задачи
- Пройти все `admin/*`, `finance/*`, `documents/*`, `staff/*` endpoint'ы:
  - добавить обязательную auth-проверку;
  - добавить permission-check на сервере.
- Закрыть публичные чувствительные route.
- Убрать fallback-secret, запускать backend только при валидном `SECRET_KEY`.

### Frontend задачи
- Сверить UI-ограничения ролей с backend-правилами (чтобы не было расхождения).

### QA задачи
- Негативные тесты:
  - пользователь без прав не может выполнить CRUD;
  - прямой вызов API без токена возвращает `401/403`.

**Definition of Done:**
- все критичные route защищены сервером;
- тест-кейсы на запрет доступа проходят.

---

## Этап 2: Целостность данных (P0, 1-2 дня)

**Цель:** исключить частичные и несогласованные записи.

### Backend задачи
- Сделать атомарные транзакции для:
  - документов поступления,
  - операций закрытия брони,
  - влияния на склад и финансы.
- Зафиксировать бизнес-правила update/delete для документов:
  - как корректируется склад,
  - как корректируется финконтур.
- Добавить защиту от редактирования исторически критичных полей без спец-сценария.

### QA задачи
- Тесты на rollback:
  - ошибка на шаге N не оставляет “половину” документа.
- Тесты на консистентность:
  - склад и финансы совпадают с документами.

**Definition of Done:**
- все критичные сценарии проходят в транзакции;
- нет частичных данных после искусственных ошибок.

---

## Этап 3: Минимальный quality gate (P0, 1 день)

**Цель:** получить обязательный минимум автопроверок перед запуском.

### Тестовый минимум
- Auth smoke: login/refresh/logout.
- RBAC smoke: ключевые запреты по ролям.
- Business smoke:
  - создание/изменение/закрытие брони,
  - документ поступления,
  - попадание данных в финансы.

### CI минимум
- Запуск lint + tests на каждый PR.
- Блок merge при падении тестов.

**Definition of Done:**
- CI pipeline зеленый;
- smoke-тесты стабильны и воспроизводимы.

---

## Этап 4: Миграции и релизный контур (P1, 1-2 дня)

**Цель:** сделать предсказуемый деплой.

### Backend/DevOps задачи
- Единый подход к миграциям (без ручного дрейфа схемы).
- Добавить pre-deploy check миграций.
- Добавить:
  - `/health`,
  - `/readiness`.
- Настроить централизованные структурные логи.

### Release задачи
- Описать runbook:
  - порядок деплоя,
  - rollback шаги,
  - проверка после релиза.

**Definition of Done:**
- есть рабочий runbook;
- деплой можно повторить по инструкции без “ручной магии”.

---

## Этап 5: Предрелизная стабилизация (P1/P2, 1-2 дня)

**Цель:** снизить риск инцидентов в первые дни после запуска.

- Провести ручной regression pack:
  - бронирование,
  - документы,
  - финансы,
  - роли.
- Провести легкий load-smoke на ключевых экранах и API.
- Включить мониторинг ошибок и алерты.

**Definition of Done:**
- нет критических дефектов класса blocker;
- есть список известных ограничений и обходов.

---

## Этап 6: Soft Launch (0.5-1 день)

**Цель:** запустить платформу на ограниченную аудиторию.

- Ограничить контур пользователей (админы/директор + 1-2 оператора).
- Наблюдать 24-48 часов:
  - ошибки,
  - производительность,
  - качество данных.
- Ежедневный triage и hotfix только по P0/P1.

**Definition of Done:**
- нет критических сбоев;
- данные по складу/финансам сходятся с фактом.

---

## Этап 7: Полный запуск (Go-Live)

**Условие перехода:** пройдены все пункты launch-gate.

- Открыть доступ всем целевым ролям.
- Запустить post-launch мониторинг на 7 дней.
- Зафиксировать backlog улучшений (P2) в отдельный спринт.

---

## Launch Gate (чек-лист перед Go-Live)

- [ ] Все P0 задачи закрыты и приняты QA.
- [ ] Проверки RBAC на backend покрывают критичные route.
- [ ] Документы/бронирования/финансы проходят тесты консистентности.
- [ ] CI стабилен (lint + tests + миграции).
- [ ] Есть backup + тест восстановления.
- [ ] Есть runbook деплоя и rollback.
- [ ] Soft launch прошел без blocker-инцидентов.

---

## Рекомендуемый календарный план

- **День 1:** Этап 0 + Этап 1
- **День 2:** Этап 2
- **День 3:** Этап 3 + начало Этапа 4
- **День 4:** завершение Этапа 4 + Этап 5
- **День 5:** Этап 6 (soft launch) + решение о Go-Live

Если команда маленькая, закладывайте +2-3 дня буфера.
